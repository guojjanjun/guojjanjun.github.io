istio中的流量分析

发表于 2018-12-04 | 更新于 2018-12-06

注： 文中示例信息来自istio官方book用例.

istio中的边车代理envoy会拦截到两类流量:

1) 入流量: 其他Pod访问当前POD中业务服务的流量。入流量访问的集群以inbound开头命名:
inbound|9080|v1|ratings.default.svc.cluster.local
2) 出流量: 当前POD中业务服务访问其他服务的流量,包括其他POD中的业务服务和istio系统相关的服务。入流量访问的集群命
名以outbound开头: outbound|9080|v1|ratings.default.svc.cluster.local

入流量的上游集群比较简单，一般只有一个：

“hosts”: [
{
“socket_address”: {
“address”: “127.0.0.1”,
“port_value”: 9080
}
}
]
这个Endpoint(127.0.0.1:9080)指向本地服务。出流量的上游集群包括所有的服务集群，因为当前POD的本地服务有访问任何服
务的可能。

Envoy代理拦截到出流量之后,会根据被访问service的域名，URL等相关信息定位到特定的路由，然后到特定的集群，再按集群内
部的负载算法获取特定的Endpoint地址，将流量发送给这个Endpoint。

这个复杂的过程中,有两个重要的参数:

litener.use_original_dst:
如果采有iptables重定向连接,那么envoy接受连接的监听端口与连接请求的原始地址是不同的。如果listener该属性设置为true，该listener将接收到的连接较交给符合请求原始地址的listener.由新的listener接受和处理该连接请求。前一个listener相当
于分发连接。在istio中，envoy的命名为virtual的listener即是如此。到达envoy的连接统一重定向到命名为virtual的
listener(监听地址为127.0.0.1:15001)，由该listener来选择合适的listener处理连接。

litener.bind_to_port
这个属性在api-v2中找不到了，也没找到替代者的说明。在istio 1.0版本中仍然使用他，虽然标记已过期。该属性表示当前
listener是否绑定端口。不绑定的端口的listener，如何接受连接？由标识use_original_dst的listener
(上文中的名为virtual的listener)接受连接，并转发给他。示例中0.0.0.0_9080这个Listener接手了所有9080端口的出流量。

[root@master ~]# kubectl exec productpage-v1-f5fcbd489-s7tfz -c istio-proxy – netstat -anp
tcp 0 0 0.0.0.0:9080 0.0.0.0: LISTEN -
tcp 0 0 127.0.0.1:15000 0.0.0.0: LISTEN 13/envoy
tcp 0 0 0.0.0.0:15001 0.0.0.0:* LISTEN 13/envoy
9080本地服务的端口，15000是envoy代理的管理端口，0.0.0.0:15001就是virtual listener的监听地址，envoy用来接收流量。
在evnoy的配置中看到其他listener都没有绑定端口。
