istio中的流量简介

发表于 2018-12-04 | 更新于 2018-12-05

istio中的边车代理envoy会拦截到两类流量:
1) 入流量: 其他Pod访问当前POD中业务服务的流量。入流量访问的集群命名以inbound开头:
inbound|9080||ratings.default.svc.cluster.local
2) 出流量: 当前POD中业务服务访问其他服务的流量,包括其他POD中的业务服务和istio系统相关的服务。入流量访问的集群命名
以outbound开头: outbound|9080||ratings.default.svc.cluster.local

入流量的上游集群比较简单，一般只有一个：

“hosts”: [
{
“socket_address”: {
“address”: “127.0.0.1”,
“port_value”: 9080
}
}
]
这个Endpoint(127.0.0.1:9080)指向本地服务。9080是istio官方book用例所有服务采用的端口。
出流量的上游集群包括所有的服务集群，因为当前POD的本地服务有访问任何服务的可能。

Envoy代理拦截到出流量之后,会根据被访问service的域名，URL等相关信息定位到特定的路由，特定的集群，再按集群内部的负载算法获取特定的Endpoint地址，将流量发送给这个Endpoint。这个复杂的过程中,有两个重要的参数:

litener.use_original_dst:
如果采有iptables重定向连接,那么envoy接受连接的监听端口与连接请求的原始地址是不同的。如果listener该属性设置为true，该listener将接收到的连接较交给符合请求原始地址的listener.由新的listener接受和处理该连接请求。前一个listener相当于分发连接。在istio中，envoy的命名为virtual的listener即是如此。到达envoy的连接统一重定向到命名为virtual的listener
(监听地址为0.0.0.0:15001)，由该listener来选择合适的listener处理连接。

litener.bind_to_port
这个属性在api-v2中找不到了，也没找到替代者的说明。在istio中仍然有用他，虽然标记已过期。该属性表示当前listener是否
绑定端口。不绑定的端口的listener，如何接受连接？原因是use_original_dst的listener代为接受连接，并转发给他。
0.0.0.0_9080这个Listener接手了所有9080端口的出流量。

集群名:inbound/outbound

正常情况下inbound是入流量访问的服务集群,outbound集群是出流量访问的集群。inbound集群很简单，就一个成员监听在
127.0.0.1地址，端口就是本地业务服务监听的端口。这个集群的endpoint就是本地业务服务。
outbound集群是本地业务服务可能要访问的服务集群。
